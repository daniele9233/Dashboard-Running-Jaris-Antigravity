# METIC LAB AI Context

Last curated context update: 2026-05-04

## Product In One Paragraph

METIC LAB is a sport-tech dashboard for running performance. It ingests Strava runs and Garmin running dynamics, computes training state, VDOT, race predictions, fatigue/freshness, biomechanics and Runner DNA, then turns those signals into dashboard cards, analytics, a training plan, and coach-like insight.

## Runtime Architecture

- The frontend is a Vite React SPA. `src/App.tsx` owns the top navigation and routes.
- The API layer is centralized in `src/api/client.ts` and `src/api/index.ts`.
- Shared API/domain types live in `src/types/api.ts`.
- The backend is currently a large FastAPI-style monolith in `backend/server.py`.
- MongoDB collections are accessed directly from `backend/server.py` through Motor.
- The frontend points at `VITE_BACKEND_URL`; if absent it falls back to the Render backend URL.

## Main Frontend Routes

- `/`: dashboard.
- `/activities`: activities list with map.
- `/activities/:runId`: run detail.
- `/training`: training calendar and plan generation.
- `/statistics`: analytics/statistics shell.
- `/runner-dna`: original Runner DNA.
- `/runner-dna-v2`: premium real-data Runner DNA V2.
- `/profile`: profile.
- `/recovery`, `/biometrics`, `/insights`: currently placeholders/coming soon in routing.

## Data Sources

- Strava is the main activity source. Sync happens through `/api/strava/*`.
- Strava is local multi-athlete: `strava_tokens` can store multiple athletes, one token has `active: true`, and all current reads/syncs use the active athlete.
- Garmin can provide running dynamics through auth/sync endpoints and through manual CSV import.
- Garmin CSV rows are stored separately, deduped/repaired, linked to matching Strava runs, then used to enrich biomechanics.
- Fitness/freshness is computed from run load and projected forward over days without running.
- Runner DNA combines Strava history, fitness/freshness, best efforts, and Garmin dynamics.

## Critical Invariants

- Never expose secrets or env values in context files.
- Keep every backend user query scoped by `athlete_id`.
- Do not reintroduce "latest Strava token wins"; use `_get_active_strava_token()` and the active athlete endpoints.
- Local multi-athlete Strava is not full SaaS auth. `user_id`/JWT isolation is still only scaffolded.
- Preserve Render memory constraints: list endpoints must not include full streams.
- Cadence should be normalized once into steps per minute (`spm`).
- Garmin CSV cadence is authoritative when matched.
- Runner DNA V2 should not invent values. Missing data must show `N/D` or a clear unavailable state.
- Generated `repo-map.md` and `repo-map.json` are deterministic and should be updated by `npm run context:update`.

## Feature Areas

### Dashboard

The dashboard combines readiness, TSB/CTL/ATL, VDOT, fatigue, race predictions, weekly/monthly/yearly volume charts, best efforts, last run map and server connectivity state. Main file: `src/components/DashboardView.tsx`.

### Activities

Activities show runs, map context, cadence, HR, drift, plan feedback, route detail and Live Telemetry 3D. Main files: `src/components/ActivitiesView.tsx`, `src/components/RoutesView.tsx`, `src/components/LiveTelemetry3DMap.tsx`.

### Training

The training page renders the annual/monthly calendar, plan controls, week sidebar and plan generation/adaptation flows. Backend plan logic is in `backend/server.py` under `/api/training-plan/*`. Plan generation returns `history_context` and adapts VDOT, phase allocation and volume to stop days, recent volume, quality history and CTL/ATL/TSB.

### Statistics

Statistics is a multi-tab advanced analytics experience. The most active files are in `src/components/statistics/`, especially `StatisticsView.tsx`, `AnalyticsV2.tsx`, `AnalyticsV3.tsx`, `AnalyticsV4.tsx`, `AnalyticsV5.tsx`.

### Runner DNA

`/runner-dna` is the original profile view. `/runner-dna-v2` is a premium UI driven by the same real backend data through `useRunnerDnaUiModel()` and `buildRunnerDnaUiModel()`.

### Garmin Import

Garmin CSV import/link is backend-heavy. It validates and normalizes cadence, GCT, vertical oscillation, vertical ratio, stride length, HR, pace and power. Import/link invalidates analytics and Runner DNA caches.

## Current Known Sensitive Areas

- `backend/server.py` is large and mixes many domains. Before editing, locate exact route/helper with repo-map or search.
- Runner DNA cache signature and data freshness are important. Do not reduce it to only last Strava date.
- Cadence has had doubled/halved bugs. Use canonical helpers.
- Mapbox/3D pages can be performance sensitive if too many polylines/geometries are rendered at once.
- The repo currently has local/legacy folders (`restyling`, `.design-ref`) that may not be production paths.
